ESE 2011 Team 5 Code Review

by ESE2011 Team 3

Review

The following section contains our coding review about the application ESECalendar which has been coded by ESE2011's team number 5.

Design

  • There's only one controller - Application - which is very big and thus unreadable. If all the other methods concerning users are in Application, also the Register#createNewUser() should be placed there.
  • The logic of creating the grid of the calendar is really inconsistent. Starting the application in November causes the calendar to have 30 days in every single month. Starting it in December implies 31 days for each month. In conclusion: the calculation you intended to do does not work at all.
  • You have a nice object structure, which is starting from User.
  • Why isnt your ESEDatabase an ordinary object? It could be easily adapted to an object. In that case you would extend the object structure, so that the ESEDatabase would be the root node.
  • The Security controller is pretty strange. You redefine the static method Security#invoke() from the play! framework in order to implement a redirection! What happens if the programmers of play change Security#invoke() method in a later release?

Coding Style

  • Almost every setter or getter sticks to the JavaBeans Conventions. That's good!
  • Frequent use of meaningless names for either methods or variables (constants and locals).
  • Class names are all started by ESE which helps destroying readability.
  • Almost every method throws an ESEException and there is even some which are declared as throwing one but actually don't. Further, the exceptions are mostly cascaded up the call stack to the play implementation and not handled at all.
  • The coding style isn't consistent. Requesting a user id is done by User#getUserID(). Requesting a calendar id should be therefore: Calendar#getCalendarID(), but is Calendar#getID().
  • It's okay to name getters for id's or names after the JavaBeans Conventions. But if you ever need an interface for events, calendars or users then you're better off by naming the attributes as id or name and the getters as getId() or as getName().

JavaDoc

  • In general, the JD isn't infmormative enough. You state the obvious and leave out responsibilities and usage of the parameters as an example.

Tests

  • The tests are helpful to see how the model objects are interacting.
  • They document the usage of calendar, event or user objects.
  • The unerstanding of the calendar test made it easy to implement the event search functionality.
