ESE 2011 Team 5 Code Review

by ESE2011 Team 3

Review

The following section contains our coding review about the application ESECalendar which has been coded by ESE2011's team number 5.

Design

  • There's only one controller - Application - which is very big and thus unreadable. If all the other methods concerning users are in Application, also the Register#createNewUser() should be placed there.
  • The logic of creating the grid of the calendar is really inconsistent. Starting the application in November causes the calendar to have 30 days in every single month. Starting it in December implies 31 days for each month. In conclusion: the calculation you intended to do does not work at all.

Coding Style

  • Frequent use of meaningless names for either methods or variables (constants and locals).
  • Class names are all started by ESE which helps destroying readability.
  • Almost every method throws an ESEException and there is even some which are declared as throwing one but actually don't. Further, the exceptions are mostly cascaded up the call stack to the play implementation and not handled at all.

JavaDoc

  • In general, the JD isn't infmormative enough. You state the obvious and leave out responsibilities and usage of the parameters as an example.
